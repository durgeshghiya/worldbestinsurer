## Data Collection Notes

### Data Quality Assessment Framework

When collecting data from the above sources, Zura should assess each data point on:

| Criterion | Description | Rating Scale |
|---|---|---|
| **Accuracy** | Does the data match official/regulatory filings? | High / Medium / Low |
| **Freshness** | How recently was the data updated? | Current / Quarterly / Annual / Stale |
| **Completeness** | Are all fields available for all insurers? | Complete / Partial / Sparse |
| **Accessibility** | Is the data available via API, structured page, or PDF? | API / Structured HTML / PDF / Manual |
| **Legal Clarity** | Can this data be displayed on a comparison platform? | Clear / Ambiguous / Restricted |

### Data Collection Method Recommendations

| Data Type | Recommended Method | Notes |
|---|---|---|
| **Premium Quotes** | Insurer APIs or manual calculation tools | Real-time quotes require API partnerships or broker platform access. Scraping is legally risky. |
| **Product Features** | Manual curation from official product pages and policy wordings | Most reliable method; feature data changes infrequently |
| **Claim Settlement Ratios** | IRDAI Annual Report (primary), insurer websites (secondary) | Annual updates; use the regulatory source as authoritative |
| **Network Hospitals** | Insurer websites or API partnerships | Changes frequently; needs regular refresh |
| **Policy Wordings** | Direct download from insurer websites | Store locally; link to official source |
| **Solvency Ratios** | IRDAI quarterly disclosures, insurer filings | Quarterly updates from stock exchange filings |
| **Customer Reviews/Ratings** | Google Reviews, app store ratings, IGMS grievance data | Supplement with proprietary user feedback |

### Legal and Compliance Considerations for Data Use

1. **Do not scrape insurer websites** without explicit permission. This can violate terms of service and potentially the IT Act.
2. **IRDAI public data** (annual reports, regulations, circulars) is freely usable with proper attribution.
3. **Premium data** should be clearly marked as "indicative" unless sourced through an official API partnership.
4. **Policy wordings** are public documents but should link to the official source rather than hosting copies (to ensure users always see the latest version).
5. **Claim settlement ratios** from IRDAI are public data and can be displayed freely with attribution.
6. **Insurer logos and brand names** require care; use for factual comparison purposes under fair use principles, but confirm with legal counsel.
7. **Network hospital data** changes frequently; if displaying it, clearly timestamp the data and link to the official insurer source.

### API and Partnership Opportunities

| Type | Description | Effort |
|---|---|---|
| **Insurer API Partnerships** | Direct integration with insurer systems for real-time quotes and policy issuance | High effort, requires broker license and individual insurer agreements |
| **Aggregator APIs** | Integration through existing aggregator platforms | Medium effort, revenue sharing required |
| **IRDAI Data API** | No public API currently available; data must be extracted from PDFs and reports | High manual effort for initial setup |
| **IIB Data Access** | May require formal application for detailed data access | Medium effort, institutional process |

### Recommended Data Refresh Schedule

| Data Category | Refresh Frequency | Priority |
|---|---|---|
| Premium rates / calculators | Real-time or daily (via API) | Critical |
| Product features and benefits | Monthly review, update on change | High |
| Claim settlement ratios | Annual (upon IRDAI report release) | High |
| Solvency ratios | Quarterly | Medium |
| Network hospital lists | Monthly | Medium |
| Policy wordings | On product launch/revision | Medium |
| Customer reviews | Weekly aggregation | Low-Medium |
| Regulatory updates | Weekly monitoring | High |
